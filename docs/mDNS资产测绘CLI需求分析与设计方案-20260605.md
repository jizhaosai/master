## 1. 需求背景与目标

### 1.1 原始需求

使用 Golang 开发一个网络资产测绘 CLI 程序，核心能力：

- 输入：**IP 网段（CIDR）** + **端口范围**
- 输出：该网段、该端口范围内通过 **mDNS 协议**发现的资产信息
- 字段要求（最低）：`ip`、`port`、`host`、**深度识别 banner**
- banner 识别深度需达到示例所示水平（解析出 `Name / IPv4 / IPv6 / Hostname / TTL` 以及 TXT 记录中的 `model / fwVer / path / accessType` 等业务键值对）

### 1.2 目标拆解

从一名资深「协议 / IP 资产识别」工程师视角，该需求的本质是：

> **在指定 IP 范围内，主动 + 被动地采集 mDNS（Multicast DNS）/ DNS-SD（DNS Service Discovery）广播与响应报文，重组出"主机 → 服务实例 → 端口 → 属性"的资产视图，并对 TXT 记录做深度解析形成可读 banner。**

需要明确的是：**mDNS 的"端口"语义与传统 TCP 端口扫描不同**（详见第 3 章）。需求中的"端口范围"不是逐端口探测 mDNS，而是对发现到的服务实例按其 SRV 记录中声明的服务端口进行筛选。这是本方案的关键设计前提，将在协议分析章节展开论证。

## 2. 示例输出逐行解读（理解 banner 深度的关键）

需求中给出的示例是一台 QNAP NAS（`slw-nas`）的 mDNS 资产指纹。逐段拆解可反推出我们必须实现的解析深度：

```text
services:
9/tcp workstation:
    Name=slw-nas [24:5e:be:69:a3:13]      # 服务实例名（workstation 服务的实例名常含 MAC）
    IPv4=x.x.x.x                          # A 记录
    IPv6=fe80::265e:beff:fe69:a313        # AAAA 记录
    Hostname=slw-nas.local                # SRV target，即主机名
    TTL=10                                 # 记录 TTL
5000/tcp http:
    ...
    path=/                                 # TXT 记录键值：HTTP 服务路径
445/tcp smb:
    ...
5000/tcp qdiscover:
    ...
    accessType=https,accessPort=86,model=TS-X64,displayModel=TS-464C,fwVer=5.2.9,fwBuildNum=20260214   # TXT 深度键值
device-info:
    ...
    model=Xserve                           # _device-info._tcp 的 model（设备类型指纹）
548/tcp afpovertcp:
    ...
answers:
PTR:
    _workstation._tcp.local
    _http._tcp.local
    _smb._tcp.local
    _qdiscover._tcp.local
    _device-info._tcp.local
    _afpovertcp._tcp.local
```

### 2.1 从示例反推的数据模型要素

| 示例片段 | mDNS / DNS-SD 记录类型 | 资产字段 |
| --- | --- | --- |
| `9/tcp workstation` | PTR + SRV | 服务类型、端口、传输协议 |
| `Name=slw-nas [...]` | 服务实例名（PTR 的 rdata） | 实例名 instance_name |
| `IPv4=` | A 记录 | ipv4 |
| `IPv6=` | AAAA 记录 | ipv6 |
| `Hostname=slw-nas.local` | SRV.target | host |
| `TTL=10` | 资源记录 TTL | ttl |
| `path=/`、`model=...`、`fwVer=...` | TXT 记录 | banner（键值对集合） |
| `answers: PTR:` 列表 | `_services._dns-sd._udp.local` 的 PTR 应答 | 服务类型清单 |

### 2.2 结论：banner 深度 = TXT 记录完整解析 + 多记录关联

"深度识别 banner"要做到示例水平，必须：

1. 解析 PTR → 拿到服务实例列表；
2. 解析 SRV → 拿到 host + port + priority + weight；
3. 解析 A / AAAA → 拿到 IPv4 / IPv6；
4. **完整解析 TXT → 拆出全部 key=value 业务字段**（这是 banner 深度的核心）；
5. 将上述记录按 `host` 与 `实例名` 关联重组为一条资产。

单纯抓一个端口的响应字符串远远不够，必须做 **DNS-SD 记录关联重组**。

## 3. mDNS / DNS-SD 协议分析（设计的理论基础）

### 3.1 协议定位

- **mDNS（RFC 6762）**：在本地链路（link-local）上做无中心 DNS 解析的协议。使用组播地址 **224.0.0.251（IPv4）/ ff02::fb（IPv6）**，固定端口 **UDP 5353**。报文格式复用标准 DNS 报文结构。
- **DNS-SD（RFC 6763）**：在 mDNS 之上定义"服务发现"约定，靠 PTR / SRV / TXT 三类记录组合表达"有哪些服务、在哪个主机哪个端口、附带哪些属性"。

### 3.2 关键事实：mDNS 通信只发生在 UDP 5353

这是整套方案最重要的协议前提：

> 无论目标服务是 `http(5000)`、`smb(445)`、`afpovertcp(548)`，**mDNS 的查询与应答报文本身永远走 UDP 5353 组播 / 单播**。示例里的 `9/tcp`、`445/tcp`、`5000/tcp` 是 **SRV 记录中声明的"服务实际监听端口"**，不是 mDNS 报文的传输端口。

因此需求里的"端口范围"有两种可能解读，方案需同时支持并以前者为默认：

| 解读 | 含义 | 实现方式 |
| --- | --- | --- |
| **A. 服务端口过滤（默认）** | 对发现到的服务实例，按其 SRV 端口落在 `[portStart, portEnd]` 内做筛选输出 | mDNS 发现 + 端口字段过滤 |
| B. 传输端口探测 | 向目标的指定端口发送 mDNS 单播查询（RFC 6762 §5.5 允许单播 5353，但非标准端口几乎无设备响应） | 仅在 5353 上做单播探测，其余端口忽略 |

> 设计采用 **解读 A 为主**：先用标准 mDNS/DNS-SD 把网段内资产发现完整，再用端口范围对"服务端口"做过滤。这既符合协议现实，又能满足"输出 ip/port/host/banner"的诉求。

### 3.3 两种采集模式

| 模式 | 原理 | 优点 | 局限 |
| --- | --- | --- | --- |
| **组播发现（推荐）** | 向 224.0.0.251:5353 发送 `_services._dns-sd._udp.local` PTR 查询（meta-query），监听全网段回应 | 一次查询覆盖整个链路，命中率高 | 仅限本地链路（mDNS 不跨三层路由） |
| **单播查询（补充）** | 对网段内每个 IP 直接发 UDP 5353 单播 mDNS 查询（QU bit） | 可对特定 IP 定向探测，部分跨网段场景可用 | 多数设备只回组播；遍历大网段慢 |

CLI 默认两种模式都做：先组播全量发现，再对网段内未响应的 IP 做单播补探，最后用 IP 网段 + 端口范围过滤结果。

### 3.4 发现流程（DNS-SD 三段式）

```
① meta-query:  PTR  _services._dns-sd._udp.local        → 得到服务类型列表（如 _http._tcp.local）
② service-query: PTR  _http._tcp.local                   → 得到服务实例列表（如 slw-nas._http._tcp.local）
③ resolve:     SRV/TXT 实例名 + A/AAAA host              → 得到 host:port + 属性 + IP
```

实际工程中可把 ①②③ 的查询合并批量发出，并持续监听一段时间（mDNS 是异步、可重复应答的），在监听窗口内把所有记录收齐后统一重组。

## 4. 需求分析报告

### 4.1 功能性需求（FR）

| 编号 | 需求 | 说明 | 优先级 |
| --- | --- | --- | --- |
| FR-01 | CIDR 输入解析 | 支持 `192.168.1.0/24`、单 IP、IP 列表 | P0 |
| FR-02 | 端口范围输入解析 | 支持 `1-65535`、`80,443,5000`、混合 `80,5000-5100` | P0 |
| FR-03 | mDNS 组播发现 | 向 5353 组播发 DNS-SD meta-query 并监听 | P0 |
| FR-04 | mDNS 单播补探 | 对网段内 IP 发单播查询 | P1 |
| FR-05 | DNS-SD 记录重组 | PTR/SRV/TXT/A/AAAA 关联成资产 | P0 |
| FR-06 | TXT 深度解析 | 拆出全部 key=value（banner 深度核心） | P0 |
| FR-07 | 网段 + 端口过滤 | 按输入范围筛选输出资产 | P0 |
| FR-08 | 多格式输出 | text（示例同款）/ json / 写入 MySQL | P0 |
| FR-09 | 结果持久化 | 扫描任务 + 资产 + 服务记录入库 MySQL 8.0 | P0 |
| FR-10 | 任务管理 API | Gin 提供任务发起 / 查询 / 结果导出 REST 接口 | P1 |
| FR-11 | 并发与超时控制 | 监听窗口、并发数、单播超时可配 | P0 |
| FR-12 | 网卡指定 | 多网卡环境可指定出口 interface | P1 |

### 4.2 非功能性需求（NFR）

| 编号 | 需求 | 指标 |
| --- | --- | --- |
| NFR-01 | 性能 | /24 网段组播发现 + 重组 ≤ 默认监听窗口（5s 可配），单播补探并发 ≥ 256 |
| NFR-02 | 兼容性 | Windows / Linux / macOS 均可编译运行（Go 跨平台） |
| NFR-03 | 健壮性 | 单个 IP / 报文解析失败不影响整体任务 |
| NFR-04 | 可观测性 | 结构化日志（zap），关键阶段耗时统计 |
| NFR-05 | 可配置 | YAML 配置 + CLI flag 双通道，flag 优先 |
| NFR-06 | 安全 | 数据库凭据走环境变量 / 配置文件，不硬编码 |

### 4.3 边界与非目标

- mDNS 受协议限制**只在本地链路有效**，跨三层路由的网段无法通过组播发现（文档需向用户明示）；
- 不实现对非 5353 端口的"伪 mDNS"探测（无实际意义）；
- 不做主机存活的全端口扫描（那是 nmap 的范畴，超出 mDNS 资产测绘边界）。

### 4.4 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 防火墙拦截组播 | 发现不到资产 | 提供单播补探 + 明确提示 |
| 多网卡选错出口 | 收不到应答 | `--interface` 指定 + 自动枚举活动网卡 |
| 设备应答延迟 / 丢包 | 资产不全 | 多次查询 + 可配监听窗口 + 去重合并 |
| Windows 组播权限 | 绑定失败 | 文档说明以管理员运行 + 复用端口选项 |

### 5.2 核心处理流水线

```
输入(CIDR+端口范围)
      │
      ▼
[1] TargetGen 展开 IP 列表 + 解析端口集合
      │
      ▼
[2] mDNSProber 打开 UDP5353 组播 socket
      │  ├─ 发送 meta-query (_services._dns-sd._udp.local PTR)
      │  ├─ 发送各服务类型 PTR query
      │  └─ 单播补探未响应 IP
      ▼
[3] 监听窗口内收包 → RecordParser 解析为 ResourceRecord
      │
      ▼
[4] AssetBuilder 按 host/instance 关联重组
      │  ├─ PTR  → 服务类型/实例
      │  ├─ SRV  → host:port
      │  ├─ A/AAAA → ipv4/ipv6
      │  └─ TXT  → banner 键值对
      ▼
[5] Filter 按 IP 网段 + 端口范围过滤
      │
      ▼
[6] 输出: text / json / MySQL 持久化
```

### 5.3 技术选型

| 关注点 | 选型 | 理由 |
| --- | --- | --- |
| 语言 | Go 1.24.0 | 与本地版本一致，跨平台、并发友好 |
| CLI 框架 | spf13/cobra + spf13/viper | 子命令 + 配置一体化，社区事实标准 |
| Web 框架 | gin-gonic/gin | 需求指定，高性能 |
| DNS 报文 | miekg/dns | 业界最成熟的 DNS 报文编解码库，可直接构造/解析 mDNS 报文 |
| mDNS 高层（可选） | hashicorp/mdns 或 grandcat/zeroconf | 可作发现层加速；本方案以 miekg/dns 自研收发为主以获得完整 TXT/TTL 控制 |
| ORM | gorm.io/gorm + driver/mysql | 需求指定 MySQL 8.0，GORM 开发效率高 |
| 日志 | uber-go/zap | 结构化、高性能 |
| 网络底层 | golang.org/x/net/ipv4, ipv6 | 组播 join / 指定 interface |

> 选型说明：**优先使用 `miekg/dns` 直接构造和解析 mDNS 报文**，因为只有完全掌控报文才能稳定拿到 TXT 全字段与 TTL（高层库常做了裁剪）。`zeroconf` 类库可作为对照与加速发现的备选。

### 6.2 领域数据模型

```go
// internal/model/asset.go

// Asset 资产：一台主机在一次扫描中的发现结果
type Asset struct {
    ID         uint64    `gorm:"primaryKey;autoIncrement"`
    TaskID     uint64    `gorm:"index;comment:所属扫描任务"`
    Host       string    `gorm:"size:255;index;comment:主机名 *.local"`
    IPv4       string    `gorm:"size:45;index;comment:IPv4 地址"`
    IPv6       string    `gorm:"size:64;comment:IPv6 地址"`
    MAC        string    `gorm:"size:32;comment:MAC（若实例名含）"`
    DeviceModel string   `gorm:"size:128;comment:device-info.model 设备型号"`
    Services   []Service `gorm:"foreignKey:AssetID;comment:该主机暴露的服务"`
    FirstSeen  time.Time
    LastSeen   time.Time
}

// Service 服务实例：DNS-SD 的一个 service instance
type Service struct {
    ID           uint64 `gorm:"primaryKey;autoIncrement"`
    AssetID      uint64 `gorm:"index"`
    ServiceType  string `gorm:"size:128;comment:如 _http._tcp"`
    InstanceName string `gorm:"size:255;comment:服务实例名 Name="`
    Port         uint16 `gorm:"comment:SRV 端口"`
    Protocol     string `gorm:"size:8;comment:tcp/udp"`
    Priority     uint16
    Weight       uint16
    TTL          uint32 `gorm:"comment:记录 TTL"`
    TxtRecords   string `gorm:"type:json;comment:TXT 键值对(JSON)，banner 深度内容"`
    Banner       string `gorm:"type:text;comment:格式化后的可读 banner"`
}
```

```go
// internal/model/record.go —— 解析中间态
type ResourceRecord struct {
    Name  string
    Type  uint16   // dns.TypePTR / SRV / TXT / A / AAAA
    TTL   uint32
    SrcIP string   // 应答来源 IP（单播补探时用于关联）

    // 各类型负载
    PtrTarget string            // PTR
    SrvTarget string            // SRV.target (host)
    SrvPort   uint16            // SRV.port
    SrvPrio   uint16
    SrvWeight uint16
    A         string            // A
    AAAA      string            // AAAA
    Txt       map[string]string // TXT 解析后的 kv
    TxtRaw    []string          // TXT 原始字符串
}
```

### 6.3 核心模块设计

#### 6.3.1 TargetGen（网段 / 端口展开）

职责：把 CLI 输入归一化为「待匹配 IP 集合」+「端口集合」。

```go
// 端口范围语法：  "1-65535" | "80,443" | "80,5000-5100"
func ParsePorts(spec string) (map[uint16]struct{}, error)

// CIDR/IP 语法： "192.168.1.0/24" | "10.0.0.5" | 逗号分隔
func ExpandTargets(spec string) ([]net.IP, *net.IPNet, error)
```

设计要点：
- 端口用 `map[uint16]struct{}` 便于 O(1) 过滤；
- IP 既保留枚举列表（单播补探用），也保留 `*net.IPNet`（网段归属判断用），大网段不必真展开成百万条；
- 输入校验失败给出中文错误提示。

#### 6.3.2 mDNSProber（报文收发）

职责：组播 join + 收发 mDNS 报文。基于 `miekg/dns` 构造查询、`golang.org/x/net/ipv4` 控制组播。

```go
type Prober struct {
    iface       *net.Interface
    listenDur   time.Duration   // 监听窗口
    conn4        *net.UDPConn   // bound to :5353
    pktChan     chan rawPacket  // 收到的原始包
}

// 发送 DNS-SD meta-query + 各服务类型 query（组播）
func (p *Prober) MulticastQuery(ctx context.Context, types []string) error

// 对指定 IP 发单播 QU 查询
func (p *Prober) UnicastQuery(ctx context.Context, ip net.IP, types []string) error

// 监听窗口内持续收包，输出原始报文
func (p *Prober) Listen(ctx context.Context) <-chan rawPacket
```

设计要点：
- 监听 socket 绑定 `0.0.0.0:5353` 并 `JoinGroup(224.0.0.251)`；Windows 下设置 `SO_REUSEADDR`；
- 查询报文：Header.RecursionDesired=false，Question 设 QM/QU bit（RFC6762 §5.4）；
- meta-query 名称固定 `_services._dns-sd._udp.local.` 类型 PTR；
- 收发分离：一个 goroutine 持续 ReadFrom 投递到 channel，主流程在 `listenDur` 窗口内消费；
- 支持多轮查询（默认重发 3 次，间隔递增）以对抗丢包。

#### 6.3.3 RecordParser（报文解析）

职责：`[]byte` → `[]ResourceRecord`。直接用 `dns.Msg.Unpack`，遍历 Answer/Extra/Authority section。

```go
func ParseMessage(raw []byte, srcIP string) ([]ResourceRecord, error)
```

TXT 解析是 banner 深度关键：

```go
// TXT rdata 为多条字符串，每条形如 "key=value" 或裸 flag
func parseTXT(txt []string) (map[string]string, []string) {
    kv := make(map[string]string)
    for _, s := range txt {
        if i := strings.IndexByte(s, '='); i >= 0 {
            kv[s[:i]] = s[i+1:]
        } else if s != "" {
            kv[s] = ""   // 布尔型 flag
        }
    }
    return kv, txt
}
```

#### 6.3.4 AssetBuilder（记录重组 + banner 生成）

职责：把零散记录按 `host`（SRV.target）和服务实例名关联成 Asset/Service，并生成示例同款 banner。

重组算法：

```
1. 建立索引：
   srvByInstance[instanceName] = SRV记录
   txtByInstance[instanceName] = TXT记录
   aByHost[host] = []A
   aaaaByHost[host] = []AAAA
   ptrByType[serviceType] = []instanceName
2. 遍历每个服务实例：
   service.Host = srv.target
   service.Port = srv.port
   service.IPv4 = aByHost[host]
   service.IPv6 = aaaaByHost[host]
   service.Txt  = txtByInstance[instance]
3. 按 host 聚合 service → Asset
4. 从 _device-info._tcp 的 TXT.model 提取 DeviceModel
5. 从 workstation 实例名正则提取 MAC：  /\[([0-9a-f:]{17})\]/
```

banner 生成（示例同款）：

```go
func (s *Service) BuildBanner() string {
    var b strings.Builder
    fmt.Fprintf(&b, "%d/%s %s:\n", s.Port, s.Protocol, shortType(s.ServiceType))
    fmt.Fprintf(&b, "    Name=%s\n", s.InstanceName)
    fmt.Fprintf(&b, "    IPv4=%s\n", s.IPv4)
    fmt.Fprintf(&b, "    IPv6=%s\n", s.IPv6)
    fmt.Fprintf(&b, "    Hostname=%s\n", s.Host)
    fmt.Fprintf(&b, "    TTL=%d\n", s.TTL)
    for k, v := range s.Txt {          // TXT 深度键值，如 path=/、model=TS-X64
        fmt.Fprintf(&b, "    %s=%s\n", k, v)
    }
    return b.String()
}
```

## 7. 关键实现流程（伪代码 / 核心片段）

### 7.1 扫描主流程

```go
func (s *ScanService) Run(ctx context.Context, in ScanInput) (*ScanResult, error) {
    // 1. 解析输入
    ips, ipnet, err := engine.ExpandTargets(in.CIDR)
    ports, err := engine.ParsePorts(in.Ports)

    // 2. 选网卡 + 建 prober
    iface := netutil.PickInterface(in.Interface)
    prober, err := engine.NewProber(iface, in.ListenDur)
    defer prober.Close()

    // 3. 启动监听 goroutine
    pktCh := prober.Listen(ctx)

    // 4. 组播发查询（meta-query + 已知服务类型），重发 retries 次
    go prober.MulticastQuery(ctx, engine.KnownServiceTypes)

    // 5. 单播补探（可选）
    if in.Unicast {
        go fanOutUnicast(ctx, prober, ips, in.Concurrency)
    }

    // 6. 监听窗口内收包解析
    var records []model.ResourceRecord
    timer := time.After(in.ListenDur)
    for {
        select {
        case pkt := <-pktCh:
            rrs, _ := engine.ParseMessage(pkt.Data, pkt.SrcIP)
            records = append(records, rrs...)
        case <-timer:
            goto BUILD
        case <-ctx.Done():
            goto BUILD
        }
    }
BUILD:
    // 7. 重组 + 过滤
    assets := engine.BuildAssets(records)
    assets = engine.Filter(assets, ipnet, ports)

    // 8. 持久化（可选）
    if in.SaveDB {
        s.repo.SaveAssets(ctx, taskID, assets)
    }
    return &ScanResult{Assets: assets}, nil
}
```

### 7.3 构造 mDNS 查询报文

```go
func buildQuery(name string, qtype uint16) ([]byte, error) {
    m := new(dns.Msg)
    m.Id = 0                       // mDNS 查询 ID 必须为 0 (RFC6762 §18.1)
    m.RecursionDesired = false
    m.Question = []dns.Question{{
        Name:   dns.Fqdn(name),
        Qtype:  qtype,
        Qclass: dns.ClassINET,      // 高位 QU bit 可按需置位请求单播应答
    }}
    return m.Pack()
}
// meta-query: buildQuery("_services._dns-sd._udp.local", dns.TypePTR)
```

## 11. 总结与关键结论

1. **协议本质**：需求中的"mDNS 资产测绘"= DNS-SD 服务发现 + 多记录关联重组，所有 mDNS 通信只走 UDP 5353 组播；
2. **"端口范围"的正确语义**：默认按 SRV 记录声明的服务端口做过滤，而非逐端口探测 mDNS（这是协议现实决定的，方案已论证）；
3. **banner 深度的关键**：完整解析 TXT 记录的全部 key=value 并与 PTR/SRV/A/AAAA 关联，才能达到示例水平；
4. **技术落地**：以 `miekg/dns` 自研报文收发掌控全字段，cobra 做 CLI、Gin 做 API、GORM + MySQL 8.0 做持久化，Go 1.24.0 跨平台编译；
5. **可验证性**：通过报文回放 fixture 让全链路在无设备 / 无网络环境下可回归，配合真实局域网做最终数据集验收。

> 局限提示（需向使用方明示）：受 mDNS 协议约束，组播发现仅在**本地链路**有效，无法穿透三层路由发现远端网段资产；跨网段需在目标链路就近部署探针或采用其他协议手段。
